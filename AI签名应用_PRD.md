## 1. 产品概述

### 1.1 背景

签名是个人身份的视觉符号，但绝大多数人面临两个问题：一是觉得自己原本的签名"不好看"、不够有个人风格；二是在越来越多的电子化场景（电子合同、PDF 审批、电子文档、社交资料）里，缺少一个统一、美观、可复用的电子签名。

市面上现有的电子签名工具（DocuSign、各类 PDF 签名 App）主要解决"把签名贴到文档上"的问题，但几乎不解决"如何拥有一个好看的、属于自己的签名"。本产品的差异化定位在于：**用 AI 帮用户从 0 到 1 设计出一个专属签名，教用户写出来，并能直接、合规地用于各类电子文档。**

> **首发市场决策**：首发主打**英文 / 拉丁字母签名**（覆盖英语 + 法语等场景）。模板库与生成逻辑先围绕拉丁字母构建，中文签名作为后续扩展。

### 1.2 产品定位

一句话定位：**一款用 AI 帮你设计、优化、并教你写出专属签名的工具，签名可在 App 内合规地用于各类电子文档。**

### 1.3 核心价值主张

- **设计**：不懂书法、不会设计也能拥有一个有质感的个性签名。
- **个性化**：基于自己手写笔迹润色，保留个人特征，而不是套用千篇一律的模板。
- **学得会**：对 AI 生成的签名提供笔顺教学，让"好看的签名"真正变成自己能随手写出来的签名。
- **可落地且合规**：不仅停留在"图片"，还能在 App 内直接、合规地完成电子文档签署。

### 1.4 与竞品的关键区别

| 维度 | 传统电子签名工具 | 字体生成 / 艺术签网站 | 本产品 |
|---|---|---|---|
| 是否帮你"设计"签名 | 否 | 部分（套模板）| 是（模板 + AI 微调 + 笔迹润色）|
| 是否基于个人笔迹 | 否 | 否 | 是 |
| 是否教你手写 | 否 | 否 | 是（差异化亮点，限生成签名）|
| 是否能合规签署文档 | 是 | 否 | 是（SES 起步，可升级 AES/QES）|
| 设计 + 教学 + 签署一体化 | 否 | 否 | 是（核心差异化）|

## 2. 目标用户

### 2.1 用户画像

- **职场人士**：经常签电子合同、审批文件，希望有一个体面、专业的电子签名。
- **学生 / 年轻人**：想要一个酷、有个性的签名，用于社交、笔记、个人物品。
- **自由职业者 / 创作者**：需要品牌化的个人签名（作品落款、电子发票、客户合同）。
- **"想换个签名"的普通人**：对自己现有签名不满意，但不知道怎么改。

### 2.2 核心痛点

1. "我的签名很丑 / 很幼稚，但不知道怎么写好看。"
2. "找了好看的签名，但学不会怎么写出来，平时还是写原来的。"
3. "每次签电子文档都要现场手写拍照，不统一也不方便。"
4. "在欧洲签合同，不知道我的电子签名到底有没有法律效力。"

## 3. 功能范围与优先级

产品由四个功能模块组成，构成一条主线：**获得签名 → 学会签名 → 使用签名**。

```
[功能一] 模板定制        [功能二] 手写润色
        \                    /
         \                  /
          v                v
     ┌──────────────────────────┐
     │   我的专属签名（签名库）   │
     └──────────────────────────┘
          |                  |
          v                  v
   [功能三] 手写教学      [功能四] 文档签名
   （仅限功能一生成签名）  （SES → AES/QES）
```

| 功能 | 模块 | 优先级 | MVP 是否包含 |
|---|---|---|---|
| 功能一 | 模板个性化定制 + AI 微调 | P0 | ✅ |
| 功能二 | 手写拍照上传 + AI 润色 | P0 | ✅ |
| 功能三 | 签名手写教学（仅限功能一生成的签名）| P1 | ✅（MVP 可做，技术风险已规避）|
| 功能四 | 电子文档签名（SES 起步） | P0 | ✅（MVP 做 App 内 SES 签署）|
| — | App 内 AES / QES 合规签署 | P1 | ⛔（二期，对接 QTSP）|

> 决策说明：MVP 打通"生成签名 →（可选）学着写 → App 内签 PDF（SES）"的完整闭环。**手写教学仅支持功能一生成的签名**——因为这些签名由系统按已知矢量路径生成，生成时即可记录笔顺，从而绕开"任意图片笔顺还原"这一高风险技术难题。合规等级更高的 AES/QES 通过对接第三方 QTSP 在二期实现。

## 4. 功能需求详述

### 4.1 功能一：模板个性化定制（P0）

**目标**：让不会设计的用户，通过选模板 + 自然语言微调，快速得到满意的签名。

**用户流程**
1. 输入姓名（首发支持英文 / 拉丁字母，含法语等；中文后续扩展）。
2. 浏览模板库，按风格筛选。
3. 选择模板 → 系统实时生成签名预览（建议一次生成 4~6 个候选）。
4. 对喜欢的候选进行 AI 微调。
5. 满意后保存到"我的签名库"。

**模板库分类（建议，首发以拉丁字母为主）**
- 商务专业（工整、稳重、适合合同）
- 艺术连笔（cursive / flowing、有设计感）
- 极简（线条干净、现代）
- 个性潦草（快速、随性、年轻）
- 字母组合 / 花押字（monogram、首字母艺术化）

**AI 微调能力（关键）**
支持两种微调方式，建议都做：
- **参数化微调**（滑块/按钮）：倾斜角度、笔画粗细、连笔程度、大小比例、装饰元素（下划线、花饰）。
- **自然语言微调**（输入框）：例如"再流畅一点""更商务一些""把名字最后一笔拉长"。这是产品体现"AI 能力"的核心交互点。**每次 AI 生成 / 微调将消耗积分（见第 8 节商业模式）。**

**需求要点 / 边界**
- 首发只做拉丁字母的视觉逻辑；中文签名属另一套生成逻辑，扩展时单独设计，不与拉丁字母混用。
- 生成需要有"实时感"，等待过长会显著降低体验，需关注生成耗时（见第 7 节）。

---

### 4.2 功能二：手写签名拍照上传 + AI 润色（P0）

**目标**：保留用户本人笔迹特征的前提下，把潦草/拍糊的手写签名变成干净、可用的电子签名。

**用户流程**
1. 用户在纸上手写签名。
2. 拍照或从相册上传（也可支持在手机上直接手指/触控笔书写）。
3. AI 自动处理：
   - 去除背景、纸张阴影、杂色；
   - 提取笔迹、平滑线条、补全断笔；
   - 矢量化（输出可无损缩放的 SVG）。
4. 生成多个润色候选（如"忠实原迹""略微优化""艺术加工"三档）。
5. 用户选择 → 保存到签名库。

**需求要点 / 边界**
- 必须**保留个人特征**，润色不等于"换成另一个签名"，否则失去"我的签名"的意义。建议提供"润色强度"调节。
- 处理拍照常见问题：光线不均、透视倾斜、笔迹太浅、背景有横线（笔记本纸）。
- 隐私：手写签名属于敏感个人信息，需明确数据存储与删除策略（见第 9 节）。

---

### 4.3 功能三：签名手写教学（P1，仅限功能一生成的签名）

**目标**：把 AI 生成的签名拆解成可学习的笔顺步骤，让用户能亲手写出来——把"好看但写不出来"变成"自己随手能签"。

**关键约束（也是技术上能成立的前提）**
> **仅支持功能一（模板生成）产出的签名。** 这类签名由系统按已知矢量路径生成，可在生成那一刻直接记录每一笔的顺序、走向与起收笔，**无需做任何"笔顺还原"推断**，从而绕开本功能原本最大的技术风险。
>
> 功能二（手写润色）与任意上传图片**不进入**教学模式（它们只是图像/路径，缺乏可靠的笔顺信息）。在这些签名上，"学着写"入口置灰或不展示。

**用户流程**
1. 在签名库选择一个**由功能一生成**的签名 → 进入"学写"模式。
2. **笔顺动画**：按记录的笔画顺序分步演示每一笔的起笔、走向、收笔，支持调速、单笔重播。
3. **描红练习**：屏幕叠加浅色签名引导线，用户用手指 / 触控笔描摹。
4. **临摹反馈（可选，二期增强）**：对比用户书写轨迹与目标路径，给出贴合度提示。

**数据要求**
- 功能一在生成签名时，必须**结构化保存笔画数据**（每笔的有序路径点、先后顺序），作为教学与描红的数据源。这要求生成模块从一开始就以"有序笔画集合"而非"单张图"为输出结构。

**需求要点 / 边界**
- 教学功能依赖功能一的笔画数据结构，二者需在设计阶段对齐数据格式。
- 这是体现产品"教你写"差异化的核心模块，但属于"锦上添花"，不应阻塞 MVP 核心签署闭环；可作为 MVP 内的轻量版本（笔顺动画 + 描红）先上线，临摹评分放二期。

---

### 4.4 功能四：电子文档签名（P0 起步，App 内直接签署）

**目标**：让签名真正"用起来"——用户在 App 内直接完成文档签署，并随合规等级提供不同法律效力。

#### 4.4.1 核心认知（团队必须对齐）

**好看的签名 ≠ 有法律效力的签名。** 用户设计的签名只是**视觉呈现**；法律效力来自底层的**签署过程**：身份核验 + 数字证书 + 防篡改 + 时间戳。两者是叠加关系，需分层设计。

#### 4.4.2 合规分层（基于欧盟 eIDAS）

| 等级 | 实现方式 | 法律效力 | 本产品阶段 |
|---|---|---|---|
| **SES** 简单电子签名 | App 内把签名图嵌入 PDF + 时间戳 + 操作审计记录 | 基础，适合低风险日常场景 | **MVP** |
| **AES** 高级电子签名 | 对接第三方 QTSP API，加身份核验 + PKI 防篡改 | 证据效力强，但非自动等同手写 | 二期 |
| **QES** 合格电子签名 | 对接 QTSP，基于合格证书 + QSCD | 全欧盟范围内等同手写签名 | 二期 / 按需 |

#### 4.4.3 关键实现原则

- **不自建密码学基础设施、不申请成为 QTSP**（监管严、成本高）。AES/QES 一律通过对接成熟的 QTSP 或聚合型 API（如 eID Easy、Signicat）实现，开发者无需自行处理证书与硬件设备。
- QTSP 按次计费（业内约每次签署 5~20 欧元），因此 QES 适合做"按次付费 / 按次扣积分"的高价值场景，而非默认能力。
- **关注 eIDAS 2.0 / EUDI Wallet 趋势**：欧盟成员国须在 2026 年 12 月前提供数字身份钱包，未来可让用户自带免费 QES（非商业用途）。产品中长期可规划"对接 EUDI Wallet"，降低 QTSP 成本。

#### 4.4.4 用户流程

1. 在签名库选择签名。
2. 选择用途：
   - **导出**：PNG（透明背景）/ SVG（矢量）/ PDF。
   - **App 内签 PDF**：上传 PDF → 拖拽放置签名 → 选择签署等级（MVP 仅 SES）→ 导出已签文件。
3. （二期）选择 AES/QES 时，跳转 QTSP 身份核验流程 → 完成合规签署。

**需求要点 / 边界**
- 涉及合同等正式场景，须明确告知当前签署所属等级及其法律含义，不得让用户误以为 SES 等同于 QES。
- 透明背景与高分辨率导出是基础刚需，优先保证质量。

## 5. 关键用户流程（端到端）

**新用户首次使用（黄金路径）**
```
打开 App
  → 选择获得签名的方式：[选模板] 或 [上传手写]
  → 生成 / 润色，得到候选
  → AI 微调到满意
  → 保存到签名库
  → 选择用途：[导出图片] / [学着写] / [签文档]
```

设计原则：**首次使用尽快让用户拿到一个"哇，这是我的签名"的结果**，越快到达保存这一步，留存越好。注册时赠送一批免费积分，确保用户能完整体验首次"生成"。

## 6. 非功能性需求

- **性能**：模板生成 / AI 微调单次响应建议控制在数秒内，超过需有明确的加载与进度反馈。
- **跨平台**：**移动端 + Web 双端**。移动端优先（拍照、手写、随身签署场景最强），Web 端紧随（桌面办公场景下的文档签署体验更好），尽量共享同一套 AI 与签署后端。
- **离线**：核心 AI 能力依赖云端，需处理弱网/无网时的降级提示。
- **多语言**：界面至少支持英文、中文；签名生成首发仅做**拉丁字母**逻辑（覆盖英语 + 法语），中文逻辑后续扩展。

## 7. 技术方案考虑（供研发预研）

| 功能 | 涉及技术 | 风险等级 |
|---|---|---|
| 模板生成 | 字体/路径渲染、参数化 SVG、风格化生成、**有序笔画数据输出** | 中 |
| AI 自然语言微调 | LLM 解析指令 → 映射到签名参数/重生成 | 中 |
| 手写润色 | 图像分割、去背景、笔迹提取、矢量化 | 中 |
| 手写教学 | 复用生成时的笔画数据做笔顺动画 + 描红（**不做笔顺还原**）；临摹评分（二期）| 低～中 |
| 文档签名 SES | PDF 渲染与合成、透明背景导出、时间戳与审计日志 | 低 |
| 文档签名 AES/QES | 对接 QTSP / 聚合 API（CSC API）、身份核验流程 | 中（依赖第三方）|

研发建议：
1. **生成模块从一开始就以"有序笔画集合"为输出结构**（而非单张位图），这是手写教学功能能否成立的前提，需在功能一设计阶段就定好数据格式。
2. 先用现成的图像处理 / 矢量化方案验证"手写润色"质量是否达标。
3. AI 微调可先用规则 + 模板参数实现，再逐步引入更强的生成模型。
4. AES/QES 不自建，做 QTSP 选型预研（覆盖范围、单价、是否支持 Web + 移动、是否对接 EUDI Wallet）。
5. 积分系统需在后端统一计量 AI 调用成本，便于定价与风控（防刷）。

## 8. 商业模式（AI 积分制 + 增值）

采用 **AI 积分制为主、增值能力为辅** 的混合模式。理由：AI 生成 / 微调 / 润色都有真实的按次计算成本，积分能把成本与使用量对齐；而 QES 签署本就由 QTSP 按次计费，天然适合按次扣积分。

**积分消耗（示例，待定价校准）**
- 生成一组签名候选：扣 X 积分
- 一次 AI 自然语言微调：扣 Y 积分
- 一次手写润色：扣 Z 积分
- 一次 QES 合规签署：按 QTSP 成本折算积分（高单价场景）

**获取积分**
- 注册赠送：保证新用户能完整走通首次"生成→保存→导出"。
- 充值积分包：一次性购买。
- 订阅制：月度自动补充积分 + 权益（高级模板、高清/矢量无水印导出、SES 文档签署不限次等）。

**关键设计原则**
- 签名是"一次创建、长期复用"的产品：**创建类动作**（消耗 AI）用积分计费；**SES 文档签署**这类低边际成本、高频的动作，建议归入订阅权益或低/免积分，避免"处处收费"的体验。
- **AES/QES** 因有第三方硬成本，适合按次付费 / 按次扣积分。
- 必须做**防刷**：限制免费积分滥用、异常调用风控。

## 9. 成功指标（North Star & 关键指标）

- **北极星指标**：成功保存第一个签名的用户数 / 比例。
- **激活率**：新用户中完成"生成并保存签名"的比例。
- **核心动作**：人均签名导出次数、文档签署次数。
- **变现指标**：付费转化率、ARPU、积分包/订阅购买率、积分消耗与充值比。
- **功能验证（功能三 教学）**：生成签名的用户中进入"学写"模式的比例、完成描红的比例——用来判断"教学"是否为真需求、是否值得在二期加投入。
- **留存**：次日 / 7 日留存；是否有复用签名、重复签署文档的行为。

---

## 10. 隐私与合规

- 手写签名、姓名属敏感个人信息（GDPR 下尤其受关注，团队在法国/欧盟运营需重点合规）。需明确：是否上传云端、保存时长、是否用于模型训练（默认不应用于训练，或需显式授权）。
- 提供"删除我的签名与原始照片"入口。
- 明确各签署等级（SES/AES/QES）的法律含义，不得让用户误以为低等级签名等同于手写。
- AES/QES 涉及的身份核验数据由 QTSP 处理，需在隐私政策中说明数据流向。

## 11. 范围之外（本期不做）

- 团队/企业协作签署、签署流程审批（DocuSign 类重功能）。
